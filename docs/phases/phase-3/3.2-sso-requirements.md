# GCIP Multi-Tenancy & SSO Requirements (ResiliPath)

## Overview
ResiliPath utilizes **GCP Identity Platform (GCIP)** to support enterprise SSO (SAML 2.0 and OIDC). GCIP multi-tenancy allows us to isolate IdP configurations per tenant.

## Configuration Requirements

### 1. Project-Level
- GCIP must be enabled in the GCP Console.
- Multi-tenancy must be enabled.

### 2. Tenant-Level (Isolated)
- Each ResiliPath `tenantId` maps to a GCIP `tenant`.
- **SAML:** Requires IdP Entity ID, SSO URL, and X.509 Certificate.
- **OIDC:** Requires Client ID, Client Secret (optional for PKCE), and Issuer URL.

### 3. IdP Discovery Logic
- **Domain-Based:** Discovery by email domain (e.g., `user@acme.com` -> `acme` tenant).
- **Tenant-Based:** Direct selection or URL path (e.g., `resilipath.com/t/acme`).

### 4. Attribute Mapping
- IdP attributes (e.g., `http://schemas.microsoft.com/ws/2008/06/identity/claims/groups`) must be mapped to ResiliPath roles:
    - `Admin`: `resili-admin` group
    - `User`: `resili-user` group (default)
    - `Moderator`: `resili-mod` group

### 5. Custom Claims Lifecycle
1. User logs in via SSO.
2. GCIP issues a token with IdP assertions.
3. Cloud Function trigger (`beforeUserCreated` or `onUserCreated`) extracts assertions.
4. Logic validates against `Tenant.authConfig.attributeMapping`.
5. `admin.auth().setCustomUserClaims()` is called with the derived role.

## Security Considerations
- **SSO Enforcement:** If `authConfig.enforceSso` is true, the frontend must block email/password entry for that domain.
- **Auditing:** All SSO-related logins and attribute mappings must be logged to the `audit_logs` collection.
