## Summary

Enhanced the offline resilience planning experience with a robust conflict resolution system and optimized Service Worker caching. This ensures that planners can safely reconcile offline changes with the cloud while maintaining high availability of critical data.

---

## Work Completed

### 1. Conflict Resolution System
- Developed the `SyncConflictModal.tsx` component to manage data reconciliation when returning online.
- Updated the `useOfflineSync` hook with `syncToCloud` capability, allowing users to explicitly push local drafts to Firestore after an offline session.
- Implemented timestamp comparison in the Plan Editor to detect when a local draft is newer than the current cloud version.

### 2. Service Worker & Caching Optimization
- Refined `next.config.js` with Workbox runtime caching for the Firestore API (`https://firestore.googleapis.com/.*`).
- Configured a `NetworkFirst` strategy for critical data endpoints, ensuring that the PWA prioritizes fresh data but remains functional during intermittent connectivity.

### 3. Integrated Planning UX
- Seamlessly integrated the conflict resolution flow into the `PlanEditor` route.
- Ensured the `SyncStatus` UI provides granular feedback during the background sync process.

---

## Architectural Decisions

### Explicit User Consent for Sync
- Decided to require explicit user confirmation before overwriting cloud data with a local draft. This prevents accidental data loss if multiple users were editing the same plan and avoids complex automated merging that could compromise data integrity.

### Runtime API Caching
- Chose the `NetworkFirst` caching strategy for the Firestore SDK. This provides the best balance between data freshness and offline reliability for the ResiliPath SaaS model.
