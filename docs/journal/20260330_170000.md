## Scope Change Decision Log

### Original Scope (v2.0)
- Multi-module MVP: Resilience Exercise Engine + BCP Plan Generation
- MVP Launch: 05/01/2026
- Full evidence types (Photo, File, Text, Webhook)
- Full reporting exports (PDF, Excel, PowerPoint)
- Offline mode (PWA) included in MVP

### Revised Scope (v4.0 - Single-Module MVP)
- **Single-module MVP**: Resilience Exercise Execution Engine ONLY
- **MVP Launch**: 04/01/2026 (4 weeks earlier)
- **Evidence**: Photo (25MB max, JPG/PNG/WebP) + Text only; File + Webhook deferred to Phase 2
- **Reporting**: PDF export only; Excel/PPT deferred to Phase 2
- **Offline Mode**: Deferred to Phase 2; MVP is online-only PWA
- **Minimal Viable Foundation**: Placeholder tables + feature flags for future modules

### Rationale for Scope Change
| Factor | Original Approach | Revised Approach | Benefit |
|--------|------------------|-----------------|---------|
| Time-to-Market | 8-week Phase 1 | 4-week Phase 1 | 4 weeks faster launch; earlier customer validation |
| Development Risk | Multi-module complexity | Single-module focus | Reduced complexity; sharper focus on core value |
| Pilot Validation | Broad feature set | Core Resilience execution only | Clearer signal on product-market fit |
| Resource Allocation | 3.8 FTE x 8 weeks | 3.8 FTE x 4 weeks | Lower initial burn; flexibility to pivot |
| Future Extensibility | Build all features upfront | Build foundation + feature flags | "Build contracts, not features" pattern |

### Strategic Tradeoffs Accepted
| Tradeoff | Mitigation |
|----------|------------|
| Deferred BCP Plan Generation | Placeholder tables + feature flags enable seamless Phase 2 activation |
| Deferred Offline Mode | Online-only PWA sufficient for initial pilot; offline mode added based on feedback |
| Reduced Evidence Types | Photo + Text sufficient for SOC 2 audit compliance; File + Webhook added post-validation |
| PDF-Only Reporting | PDF sufficient for audit compliance; Excel/PPT added based on user feedback |

## Key Decisions Documented

### Decision 1: Minimal Viable Foundation Pattern
| Principle | Implementation |
|-----------|---------------|
| Build contracts, not features | Define API interfaces and database relationships for future modules, but hide behind feature flags |
| Document extension points | Every AResilience includes "Future Extension" section showing how to add Module 2+ without breaking changes |
| Version everything | API v1, database migration versioning, feature flag schema — all designed for evolution |
| Test the foundation | Phase 0 includes "Module 2 Simulation Test" that verifies foundation can support BCP module addition |

## Alternatives Considered (Documented for Audit)

| Decision | Selected Option | Rejected Alternatives | Rationale for Selection |
|----------|----------------|----------------------|------------------------|
| MVP Scope | Single-module (Resilience Exercise Engine) | Multi-module MVP (Resilience + BCP) | Accelerate time-to-market; reduce development risk; sharper value proposition |
